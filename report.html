low Prototype Pollution
  • Module: yargs-parser

  • Reported by: Snyk Security Team
  • CWE-471
  • Vulnerable: <13.1.2 || >=14.0.0 <15.0.1 || >=16.0.0 <18.1.2

  • Exploitability: 1

Overview

Affected versions of yargs-parser are vulnerable to prototype pollution. Arguments are not properly sanitized, allowing an attacker to modify the prototype of Object, causing the addition or modification of an existing property that will exist on all objects.
Parsing the argument --foo.__proto__.bar baz' adds a bar property with value baz to all objects. This is only exploitable if attackers have control over the arguments being passed to yargs-parser.

Findings

  • react-scripts>webpack-dev-server>yargs>yargs-parser

Remediation

Upgrade to versions 13.1.2, 15.0.1, 18.1.1 or later.

high Remote Code Execution
  • Module: serialize-javascript
  • Published: August 11th 2020
  • Reported by: Unknown
  • CWE-
  • Vulnerable: <3.1.0
  • Patched: >=3.1.0
  • Exploitability: 4

Overview

serialize-javascript prior to 3.1.0 allows remote attackers to inject arbitrary code via the function "deleteFunctions" within "index.js".

An object such as {"foo": /1"/, "bar": "a\"@__R-<UID>-0__@"} was serialized as {"foo": /1"/, "bar": "a\/1"/}, which allows an attacker to escape the bar key. This requires the attacker to control the values of both foo and bar and guess the value of . The UID has a keyspace of approximately 4 billion making it a realistic network attack.

The following proof-of-concept calls console.log() when the running eval():
eval('('+ serialize({"foo": /1" + console.log(1)/i, "bar": '"@__R-<UID>-0__@'}) + ')');

Findings

  • react-scripts>terser-webpack-plugin>serialize-javascript

Remediation

Upgrade to version 3.1.0 or later.

  • Published: October 19th 2020
  • Reported by: Alejandro Romero
  • CWE-471
  • CVE-2020-15256
  • Vulnerable: < 0.11.5
  • Patched: >=0.11.5
  • Exploitability: 5

Overview

A prototype pollution vulnerability has been found in object-path <= 0.11.4 affecting the set() method. The vulnerability is limited to the includeInheritedProps mode (if version >= 0.11.0 is used), which has to be explicitly enabled by creating a new instance of object-path and setting the option includeInheritedProps: true, or by using the default withInheritedProps instance. The default operating mode is not affected by the vulnerability if version >= 0.11.0 is used. Any usage of set() in versions < 0.11.0 is vulnerable.

Findings

  • react-scripts>resolve-url-loader>adjust-sourcemap-loader>object-path

Remediation

Upgrade to version >= 0.11.5.
